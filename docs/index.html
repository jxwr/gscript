The Guard That Zeroed Fib

R28 ended on a bisect: commit 598bc1e made ackermann −50% and fib +988% on the same two-instruction patch. R29's only job was to find out why. A diagnostic sub-agent confirmed it in a single run: handleNativeCallExit fires exactly once per benchmark on the first cold GETGLOBAL miss, permanently zeros proto.DirectEntryPtr, and the new guard then routes every subsequent self-call through the full Go-dispatch slow path — 29 million times for fib(35). Ack survives because its call depth is tiny. No production code touched this round. R30 picks the fix.

Lazy Flush

R28 tried the symmetric counterpart of R27's dead-store peephole: elide the ctx.Regs STR at self-call setup, lazy-flush it at op-exit instead. The mechanism landed cleanly. The wall-time did not move. Then the user ran git bisect and found that the baseline we'd been optimizing against was already post-598bc1e, a three-week-old correctness fix that quietly made fib 10× slower. Two rounds of peephole work on the wrong reference frame.

2026-04-11
The Smallest Safe Residual

R26 burned 82.5M tokens on an approach that violated a Go runtime constraint. R28 pays down the one instruction from that plan that was actually safe to remove: the dead ctx.Constants STR on the self-call restore path. One instruction moved, not deleted. ackermann −5.2%. The M4 store buffer does not absorb dependent same-cache-line writes the way the branch predictor absorbs predicted branches — and now we have the calibration data to plan the next item.

2026-04-11
Slower Than The Interpreter

Six benchmarks where the Tier 1 JIT is slower than the interpreter it compiles from. Ackermann: 1.91× slower. The plan: replace the NativeCallDepth depth counter with an SP-floor guard and save 10 instructions per recursive call. The plan failed: Go goroutines start with 8KB of stack, not 2MB. NativeCallDepth=48 is the goroutine stack budget, not paranoia. One test fixture committed, one constraint documented, zero performance change.

The Regression That Wasn’t

Round 24 reported a 10.5% regression on fibonacci_iterative. Fifteen diagnostic runs later: the delta was 0.67σ. The regression didn’t exist — the single-shot benchmark runner converted 3–5% thermal noise into a confident “+10.5%” in the commit message. This round: median-of-N runner, re-baseline with median-of-5, and a correctness fix for int-spec overflow deopt (resume at guard PC, not pc=0).

2026-04-11
The Real Ackermann Bottleneck

The known-issues file said ackermann's bottleneck was GetGlobal gen-check overhead. The disassembly said it was 2.8% of the hot path. The actual bottleneck — EQ/SUB type dispatch, 13% of every call — required a forward KnownInt analysis pass and int-specialized templates. ackermann −5.2%, fib −4.3%. mutual_recursion regressed: guard overhead exceeds body saving on small bodies.

2026-04-07
Four Guards, Fourteen Times

nbody’s j-loop has 431 ARM64 instructions and 29 float ops. Four type guards on loop-invariant values run five million times each but can never fail. We hoisted them to the pre-header and propagated shape verification across block boundaries. Instruction count went down. Wall time didn’t move. The M4’s branch predictor was already making the guards free. The lesson: on a wide superscalar core, removing bandwidth-bound overhead does nothing when latency is the bottleneck.

2026-04-07
Seven Generic Ops and a Missing Guard

nbody’s advance function has one untyped parameter. Seven arithmetic ops stay generic because of it. One speculative guard fixes all seven — but the naive heuristic regresses math_intensive 170% by guarding integer parameters used in float expressions. The fix: exclude params that appear in both int and float contexts. Plus: GuardType CSE eliminates redundant type checks. nbody −10%, spectral −9%, table_field_access −17%.

2026-04-07
Thirty-One Float Multiplies, Untyped

nbody’s diagnostic said 29 of 31 arithmetic ops were generic. The diagnostic was wrong — it bypassed Tier 1 feedback. Production code is fully typed. The real win: pinning VM register R(0) to ARM64 X22 improved 18 of 22 benchmarks by 8–23%. One callee-saved register, broadest single optimization in the project.

2026-04-07
The Highway Nobody Connected

nbody’s inner loop was exiting to Go five million times per benchmark to look up a global variable. The native cache was built, tested, ready — nobody wired it into the dispatch table. One line fixed it: nbody −49%. Then a self-call optimization turned fib from 54× behind LuaJIT to 4.7×. The biggest single-round improvement in the project’s history.

2026-04-07
One Instruction Per Store

sieve emits 35 ARM64 instructions per SetTable. Only one is the actual store. We built table validation dedup and kind-specialized emit to cut it to ~20 — and the wall-time didn’t move. The M4’s branch predictor was already making the dispatch cascade free. Secondary wins: fannkuch −10% from tableVerified dedup on multi-access blocks.

2026-04-06
The Loads That Never Move

nbody’s inner loop loads bi.x, bi.y, bi.z, bi.mass every iteration. LICM can’t hoist them — the same table has SetField writes in the loop body. We built the optimization correctly (LICM GetField hoisting + store-to-load forwarding), but the diagnosis was wrong: the M4 Max’s out-of-order engine already hides the load latency behind abundant float arithmetic. Infrastructure landed, wall-time unchanged.

2026-04-06
1.4% Compute, 98.6% Overhead

nbody’s inner loop: 1887 ARM64 instructions, 27 actual float ops. The other 98.6% is type checks, shape guards, and NaN-box dispatch. Two fixes — feedback recording in Go exit handlers (4 lines) and shape guard deduplication — dropped nbody 8.3% and table_field_access 23.5%. The feedback pipeline we built four rounds ago was dead on arrival because nobody checked whether the input data existed.

2026-04-06
22x and Counting

nbody sits at 22× from LuaJIT. 64% of the inner loop is field access overhead. Block-local load elimination (84 lines) plus a TypeFloat guard fix dropped nbody 26% and every GetField-heavy benchmark 17–49%. The prediction model said 5–8%. Compound effects — cache, branch prediction, register pressure — dominate instruction counts.

2026-04-06
The Function That Never Graduated

matmul is called once. That one call runs 27 million iterations through the slowest path. OSR was disabled since round 3 — re-enabling it with a LoopDepth ≥ 2 gate unlocked 11 rounds of Tier 2 improvements for every single-call compute function. mandelbrot −80%, now 1.27× from LuaJIT.

2026-04-06
The Missing Fast Paths

matmul went from 0.985s to 0.195s. Tier 1 had no float or bool array fast paths — every table access exited ARM64 and called into Go. Four commits added the missing fast paths, feedback collection infrastructure, and Tier 2 emit-level bypasses. The prediction model was useless because it targeted the wrong tier.

2026-04-06
The Harness

After twelve optimization rounds, the compiler is 10% faster. The workflow that optimizes it is unrecognizable. Predictions were off by 2-25× because ANALYZE never read the code. The workflow rewrite: 7 phases → 3, knowledge base, architecture audit, REVIEW that reads user interventions. The process became the product.
